## What Worked Well

### 1. TDD + delegate_task for IMPLEMENT
Every bug fix followed RED → GREEN through `delegate_task`. The subagent wrote the failing test first, confirmed it failed against the original code, then fixed. This eliminated speculation — we knew every fix was real before the first line of production code changed.

### 2. Honcho recall at task start
Calling `honcho_context(peer='user')` before each task creation consistently surfaced the session context. The user never had to re-explain where we were. The RECALL phase in the pipeline automated this for code-change tasks.

### 3. One delegation at a time
The user's rule of "1 delegation at a time" prevented subagent collision and made it easy to spot YAGNI violations. When a subagent over-built (phantom MODEL_REGISTRY import -> full ModelSelect component), catching it meant reverting one isolated change, not untangling three parallel threads.

### 4. Fast mode for trivial tasks
`phase_mode='fast'` saved significant overhead on test-only additions and single-line fixes. Tasks that would have taken 5 phase submissions instead completed in 3 (IMPLEMENT → VERIFY → DONE).

### 5. YAGNI discipline held up
Most fixes were 1-6 lines of source change. The architecture improvements ranged from -10 to -123 lines net. No speculative refactoring was committed. The two YAGNI skips (SQL repository, ModelRouter class) were consciously assessed and documented rather than left hanging.

## What Didn't Work

### 1. Subagent timeout on large scopes
**Problem:** Subagents dispatched with more than ~8 files (read + edit + verify cycles) consistently hit the 600-second timeout on MiniMax-M2.7. Examples: the ArchitectNote interface fix (3 files + test) and the SSE deduplication (5 files + tests) each took 150-180 seconds — near the limit. One subagent (phantom import guard) tried to build a full ModelSelect component and timed out at 600s.
**Cost:** Wasted ~10 minutes on the phantom import overbuild + revert cycle. The ArchitectNote subagent had to be re-dispatched once.
**Fix:** Keep subagent scope to 6-8 files max. Split cross-cutting changes into sequential subagents.

### 2. Pipeline scripts leaked into commits
**Problem:** Pipeline runner scripts (`.techne/scripts/submit_*.py`) and the `tasks.db` SQLite file were committed to the git history via `git add -A`. These are transient artifacts that muddy the project history.
**Cost:** 6 pipeline scripts committed to inkforge. `tasks.db` committed once to techne skill repo (later reset).
**Fix:** Add `.techne/scripts/`, `.techne/tasks/`, `.techne/memory/` to `.gitignore`. Use targeted `git add <paths>` instead of `git add -A`.

### 3. SHA gate friction in VERIFY
**Problem:** The SHA gate requires literal "passed" or "0 errors" tokens in the submission. For test-only additions where the tests already pass (no new production code), the gate blocked VERIFY because there was no diff to validate. Also, the word "failed" in any prose context triggered a false-positive failure pattern match.
**Cost:** 3 HITL cycles wasted on SHA gate re-submissions across the session.
**Fix:** Already patched: `phase_mode='fast'` skips the pass-indicator check for non-implementation tasks.

### 4. HITL re-entry state machine confusion
**Problem:** When a VERIFY submission was BLOCK_HITL'd (SHA gate), `unblock()` reset the phase to the last completed phase, but re-submitting that phase failed with "Cannot go from EVAL to VERIFY" because the unblock advanced the phase tracking differently.
**Cost:** ~5 minutes per HITL cycle debugging phase state.
**Fix:** Already patched: BLOCKED now accepts VERIFY, REVIEW, EVAL. But the fundamental confusion remains — unblock semantics could be clearer.

### 5. Pipeline scripts scattered across projects
**Problem:** Every project that uses the pipeline accumulates `.techne/scripts/` with ad-hoc runner scripts. These are duplicates of the same patterns (create task, submit phase, check status).
**Cost:** 6 scripts in inkforge, similar scripts in ms-ellen-project. Duplicate maintenance.
**Suggestion:** Ship a CLI tool with techne (`techne task create`, `techne phase submit`) instead of project-local scripts.
